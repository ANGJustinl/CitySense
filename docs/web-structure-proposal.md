# CitySense 网页构造方案

> 给导师过目用。核心讲三件事：CitySense 是什么、网页有哪些页面、每个页面长什么样。

---

## 一、产品定位

**CitySense（城市脉搏）** 是一个城市实时信号驱动的 AI 路线推荐系统。

它不解决"找个地点"的问题（高德/大众点评已经做得很好），而是解决"**今晚/这个周末，我该按什么顺序去哪几个地方**"的问题。

### 和现有产品的差异

| 维度 | 普通地图/点评 | CitySense |
|---|---|---|
| 输出 | 单个地点 | **可执行的 3 条路线**（地点 + 顺序 + 交通） |
| 数据 | POI 信息 | POI + **社交热度信号**（小红书/B站/政务）+ **实时交通**（高德 ETA） |
| 推荐 | 关键词匹配 | 多路召回 + 交通重排 + LLM 解释，**每条推荐可追溯来源** |
| 交互 | 搜索框 | 自然语言对话 + 偏好输入，AI 助手可追问 |

### 三个核心卖点

1. **路线真实可达** —— 高德 API 判断实时可达性，每段路线有真实交通耗时（不是直线距离估算）。
2. **推荐有据可查** —— 每个地点都标注数据来源（高德 POI / 小红书 / 政务公开），不是黑盒推荐。
3. **越用越懂你** —— 用户反馈（有帮助/不合适/收藏）累积成画像，后续推荐个性化。

## 二、页面结构

### 整体导航

```
首页（推荐工作台）  ←  核心，用户主要在这里
  ├─ 路线详情页      ←  点击某条路线进入
  ├─ 数据源管理      ←  admin，运营/开发用
  └─ AI 助手         ←  浮动入口，全局可用
```

**设计原则：**

- **单页为主**：用户 90% 的时间在首页工作台完成探索，不需要频繁跳转。
- **渐进披露**：首页给概览，详情页给深度信息，避免一屏塞太多。
- **AI 助手全局可用**：不占独立页面，作为浮动按钮 + 右侧抽屉，任何页面都能唤出。

### 各页面职责

| 页面 | 路由 | 职责 | 面向 |
|---|---|---|---|
| **推荐工作台** | `/` | 输入偏好 → 生成 3 条路线 → 地图可视化 → 对比选择 | 所有用户 |
| **路线详情页** | `/routes/[id]` | 单条路线的完整信息：地点、交通分段、来源信号、AI 解释 | 点击"详情"后 |
| **AI 对话助手** | 浮层（非独立页） | 自然语言问答、追问路线、查城市趋势、查个人画像 | 全局浮动入口 |
| **数据源管理** | `/admin/sources` | 查看/触发城市信号采集（小红书、政务、高德 POI） | 运营/开发 |

## 三、核心页面布局

### 1. 首页：推荐工作台（核心）

这是产品的门面，采用**地图优先的多栏布局**：

```
┌─────────────────────────────────────────────────────────────┐
│  顶部栏：CitySense logo + 导航                                │
├──────┬───────────────────────────────┬──────────┬───────────┤
│      │                               │          │           │
│ 偏好 │         地图工作区             │ 路线      │ 城市      │
│ 输入 │   （高德地图 + 3条路线         │ Inspector│ 脉冲      │
│ 栏   │    polyline + 编号 marker）    │ （当前   │ + 用户    │
│      │                               │ 选中路线 │ 画像      │
│ 兴趣 │   ┌─────────────────────────┐ │  的分数  │ explain   │
│ 心情 │   │ 指标条：候选池/ETA/      │ │  地点    │           │
│ 预算 │   │ 交通重排/Top分/缓存      │ │  来源    │           │
│ 时间 │   └─────────────────────────┘ │  信号    │           │
│      │                               │  反馈    │           │
│ 生成 │   底部：选中路线的时间轴      │  按钮    │           │
│      │   （起点→站点1→站点2→站点3）  │          │           │
│      │                               │          │           │
└──────┴───────────────────────────────┴──────────┴───────────┘
                                         ⬇ 点击某条路线
                                    [路线详情页]
```

**左侧 · 偏好输入栏**
- 兴趣标签（咖啡、展览、书店、独立音乐、夜生活…）
- 心情（Solo / 安静 / 热闹 / 约会 / 随机）
- 预算（低 / 中 / 高）
- 时间窗口（现在 / 今晚 / 周末）
- 起点（浏览器定位 / 手动地址）
- 「生成路线」按钮

**中间 · 地图工作区（视觉中心）**
- 顶部指标条：候选池数量、高德 ETA 调用数、交通重排版本、Top 路线分、缓存命中——**让评委看到推荐链路是真实的**
- 高德地图同时渲染 3 条路线（选中路线高亮 teal，其他 coral/amber）
- 编号 marker + 图例
- 底部时间轴：选中路线的站点序列 + 每段交通耗时

**右侧 · 两栏面板**
- **路线 Inspector**：当前选中路线的推荐分、总时长、地点列表、来源信号、AI 解释、反馈按钮（有帮助/不合适/收藏）
- **城市脉冲 + 用户画像**：热门标签、来源占比、交通缓存新鲜度 + 用户偏好因子 explain

**桌面端 5 列**，各列可拖拽调整宽度；**移动端纵向堆叠**（输入 → 路线选择 → 地图 → 详情）。

### 2. 路线详情页

点击某条路线后进入，展示**单条路线的完整信息**：

```
┌──────────────────────────────────────────────┐
│  ← 返回    路线标题 + 推荐分                   │
├──────────────────────────────────────────────┤
│                                              │
│              高德地图（大图）                  │
│         真实道路 polyline + marker            │
│                                              │
├──────────────────────────────────────────────┤
│  决策摘要：为什么推荐这条（AI 解释）            │
├──────────────────────────────────────────────┤
│  完整行程：                                    │
│  📍 起点                                      │
│    ↓ 约 15 min · 地铁2号线                    │
│  📍 站点1：地点名 + 标签 + 图片                │
│    ↓ 约 22 min · 71路公交                     │
│  📍 站点2：地点名 + 标签 + 图片                │
│    ↓ 约 18 min · 步行                         │
│  📍 站点3：地点名 + 标签 + 图片                │
├──────────────────────────────────────────────┤
│  来源信号：小红书热度 + 高德 POI + 政务公开     │
│  出行建议：最佳出发时间、注意事项               │
├──────────────────────────────────────────────┤
│  [有帮助]  [不合适]  [收藏]   [分享路线]       │
└──────────────────────────────────────────────┘
```

**关键设计：**
- 地图大图为主，让用户看清路线走向
- 时间轴用分段交通（真实高德耗时 + 公交线路名），不是估算
- 来源信号透明展示，体现"推荐有据可查"
- 反馈按钮直接在此页也能用

### 3. AI 对话助手（浮动浮层，非独立页）

```
                                    ┌──────────────────┐
                                    │ ✨ 城市探索助手  │
                                    │──────────────────│
                                    │                  │
                                    │ 用户：今晚静安    │
                                    │ 有什么好玩的？    │
                                    │                  │
                                    │ 🔧 查询推荐路线  │ ← 工具调用卡片
                                    │                  │
                                    │ 助手：今晚静安，  │
                                    │ 我为你推荐：      │
                                    │ 1. 路线A ...     │
                                    │ 2. 路线B ...     │
                                    │                  │
                                    │──────────────────│
                                    │ [输入框]    [发送]│
                                    └──────────────────┘
                                          ⬆ 右侧抽屉
                              ┌──────────────────┐
                              │   ✨ (浮动按钮)   │ ← 右下角入口
                              └──────────────────┘
```

**关键设计：**
- 右下角浮动按钮唤起，右侧抽屉滑出，不离开当前页面
- 流式逐字回复
- 工具调用可视化（用户看到"正在查询推荐路线"）
- 对话历史持久化（Redis 24h），刷新后可续接
- 建议问题引导（"今晚静安有什么好玩的？""最近流行什么？"）

## 四、设计语言

| 要素 | 选择 | 理由 |
|---|---|---|
| 主色 | Teal `#087f7a`（青绿） | 城市感、活力但不刺眼，区别于高德蓝/美团黄 |
| 强调色 | Coral `#c7583a`（珊瑚红） | 用于警告/负反馈/Top路线对比 |
| 圆角 | 8px | 友好但不圆滑（不是儿童感） |
| 字体 | Inter + 系统中文 | 干净、跨平台一致 |
| 信息密度 | 中高 | 工作台是工具，不是营销页，需要信息效率 |
| 交互反馈 | 轻量过渡（160ms） | 流畅但不拖沓 |

**视觉原则：让数据流动可见。** 指标条、来源信号、交通缓存新鲜度——这些不是装饰，是让用户/评委理解"这个推荐是怎么来的"。

## 五、待决问题（请导师指教）

1. **是否需要落地页？** 当前直接进工作台，新用户可能懵。是否需要一个轻量的产品介绍页作为入口？
2. **探索页 /demo /explore 如何处理？** 当前是早期 mock 原型，和真实系统脱节。是删除还是重做为"灵感发现"入口？
3. **多城市扩展节奏？** 当前硬编码上海。如果要扩城市，页面是否需要一个城市选择器入口？
4. **移动端优先级？** 工作台 5 列布局在手机上会堆叠，是否需要为移动端单独设计一个精简版（比如只保留输入 + 路线卡片列表）？
